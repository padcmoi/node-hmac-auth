# 1.1.0 ‚Äî propagation can now target the message credential store

1.0.0 introduced cross-token propagation for the **HTTP credential store**. 1.1.0 extends the same wire and the same security guarantees to the **message credential store**, behind one optional flag and one optional bridge. Everything else is unchanged.

## What is new

### 1. `propagateClientToApis` accepts `targetStore`

```ts
await hmacAuth.propagateClientToApis({
  operation: "create",
  targets: ["https://api-b.example.com"],
  clientId: "msg_amqp_orders",
  secret: "msgOrdersSecret-MO-001",
  allowedIps: ["0.0.0.0/0"],
  apiFetch: signer,
  targetStore: "message", // ‚Üê new in 1.1.0, defaults to "http"
});
```

- `targetStore: "http"` (default) ‚Äî 1.0.x behavior, no change on the wire.
- `targetStore: "message"` ‚Äî the lib reads the local `secretHash` from the **message** store (via the new `messageAuth` bridge), ships it in the same internal-management POST with `kind: "message"` in the payload, and the remote writes it to its own message store via `setSecretHash`.

### 2. `initializeHmacHttpAuth` accepts `messageAuth`

```ts
const messageAuth = initializeHmacMessageAuth({ redis, namespace, secretToken });
const hmacAuth = initializeHmacHttpAuth({
  redis,
  namespace,
  secretToken,
  internalManagementRoute: "/api/internal/hmac",
  messageAuth, // ‚Üê new in 1.1.0, optional
});
```

When `messageAuth` is provided, the single `/api/internal/hmac` route is now capable of writing to **either** credential store based on `payload.kind`. When `messageAuth` is omitted, the lib behaves exactly like 1.0.x (HTTP-only).

Required on **both ends**:

- **source** ‚Äî for the local Redis fallback to read the message-store `secretHash`.
- **target** ‚Äî for `handleInternalManagementRequest` to dispatch `kind: "message"` writes.

If a source sends `kind: "message"` to a target that did not pass `messageAuth`, the target returns `403 "Message store not configured"`.

### 3. Wire format

Payload bytes are identical to 1.0.0 for `targetStore: "http"`. For `targetStore: "message"` only one field is added:

```jsonc
{
  "clientId": "msg_amqp_orders",
  "secretHash": "<hex>",
  "allowedIps": ["..."],
  "kind": "message", // ‚Üê only present when targeting the message store
}
```

The HTTP signature of the propagation request itself is unchanged (signed by an HTTP credential the target knows ‚Äî typically `self_propagation_signer`). The "message" routing is encoded in the body, not in headers or path.

## Resolution priority (still 1.0.0 semantics, scoped to the chosen store)

When `propagateClientToApis` builds the outgoing `secretHash`:

1. Explicit `secretHash` from the caller wins.
2. Local Redis lookup on `clientId` ‚Äî in the **HTTP** store when `targetStore: "http"`, in the **message** store when `targetStore: "message"`.
3. `hashClientSecret(secret, localSecretToken)` from the caller-provided plain `secret`.

The plain `secret` is never sent on the wire.

## Out of scope

These changes apply **only** to `propagateClientToApis` and `handleInternalManagementRequest`. Everything else is unchanged:

- `signedHttpFetch`, `verifyHttpSignature`, `verifyHttpRequest` ‚Äî HTTP signing/verification unchanged.
- `signMessage`, `verifyMessage` ‚Äî message signing/verification unchanged.
- `clients.create`, `clients.setSecret`, `clients.regenerateSecret` ‚Äî still hash with the local `HMAC_SECRET_TOKEN` at write time.
- Internal-management bootstrap-then-auth ‚Äî identical.

## End-to-end proof in the POC

`poc/` was extended to demonstrate the new feature without breaking 1.0.x behavior:

- `nest-target` and `express-target` now also instantiate `initializeHmacMessageAuth` and pass it to `initializeHmacHttpAuth` via `messageAuth`. Each target exposes a public `/message/verify` route that calls `hmacMessageAuth.verifyMessage` to confirm a signature.
- `nest-source/src/microservice.cfg.ts` declares two message clients (`msg_amqp_orders`, `msg_amqp_billing`) in `hmacMessage.credentials` and adds matching `propagationPlans` entries with `targetStore: "message"`.
- `nest-source` now exposes `verifyAllPropagatedMessageClients()`: for each propagated message clientId, it `hmacMessageAuth.signMessage(...)` locally, POSTs the signed payload to each target's `/message/verify`, and logs:
  ```
  [nest_source] cross-token message verify summary ok=2/2 details=[
    {"clientId":"msg_amqp_orders", "nest_target":{"status":200,"ok":true}, ...},
    {"clientId":"msg_amqp_billing", "nest_target":{"status":200,"ok":true}, ...}
  ]
  ```
  proving each propagated message credential is verifiable on both targets despite the three services having distinct `HMAC_SECRET_TOKEN`.

The pre-existing HTTP cross-token verify (`ok=11/11`) keeps passing, demonstrating zero regression of 1.0.0 semantics.

## FAQ

> **Q ‚Äî Can one propagation plan target both stores?**
>
> No. A propagation call targets exactly one store. Run two plans if you need both.

> **Q ‚Äî Can the propagation request itself be signed by a message credential?**
>
> No. The propagation request is an HTTP POST against `internalManagementRoute`; it must be signed by an HTTP credential the remote can verify via `verifyHttpSignature` (`self_propagation_signer` or similar). The `targetStore` flag only affects where the **payload** is written on the remote.

> **Q ‚Äî Backward compatibility?**
>
> Full. 1.0.x callers don't change a line. The `messageAuth` option is opt-in and `targetStore` defaults to `"http"`.
