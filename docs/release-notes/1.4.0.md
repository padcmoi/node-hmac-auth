# Release notes — 1.4.0

Security release. Closes a bootstrap-window gap present in 1.0.0 through 1.3.0 where the credential store on every API instance accepted any first writer that POSTed to the configured `internalManagementRoute`. An attacker who reached an unprotected store before legitimate bootstrap could seed an arbitrary credential and from there sign any business request.

## What changed

The bootstrap-window lock is now always active. `requireBootstrapClientId` resolves to the federation-default value `"self_propagation_signer"` when the option is omitted, so a fresh install joins the federation safely out of the box. Overriding the option is supported only as a deliberate isolation knob; the federation-wide orchestrator (`@naskot/node-hmac-auth-management`) only pushes the canonical name.

- `feat(core): InitializeHmacHttpAuthOptions.requireBootstrapClientId defaults to "self_propagation_signer" when omitted; the bootstrap lock is always active`
- `feat(message): InitializeHmacMessageAuthOptions.requireBootstrapClientId mirrors the HTTP track default; signMessage/verifyMessage are gated by the same lock`
- `feat(http): ExpressHmacMiddlewareOptions.requireBootstrapClientId propagated through createExpressHttpHmacMiddleware and createHttpMiddlewareFactory with the same default`
- `feat(http): exported DEFAULT_PROPAGATION_KEY_CLIENT_ID constant so consumers and downstream libs reuse the federation value verbatim`

## Upgrade

The TypeScript surface is unchanged: `requireBootstrapClientId` remains optional and resolves to the secure default when omitted. Drop-in upgrade from 1.3.0 compiles and boots without code edits.

**One operational step on existing deployments**: every API instance that ran 1.0.0-1.3.0 with stored credentials but no `self_propagation_signer` will boot into the locked state after the upgrade. Push the propagation credential once to release the lock:

```bash
curl -X POST https://my-api.example.com/api/internal/hmac \
  -H "Content-Type: application/json" \
  --data '{"clientId":"self_propagation_signer","secret":"<derived-value>"}'
```

The derived value is `hmac_sha256(secretToken, "self_propagation_signer")` and is reproducible across boots without any persistent storage. The companion lib `@naskot/node-hmac-auth-management >=0.1.0` automates this step at boot via its auto-seed.

## Deprecation

Versions 1.0.0 through 1.3.0 are deprecated via `npm deprecate "@naskot/node-hmac-auth@<1.4.0"`. `npm install` against an earlier version surfaces the advisory in the CLI output.

## Compatibility

No wire change. No Redis layout change. No header change. Cross-language ports that target the canonical wire contract continue to pass the existing test vectors. Only the runtime acceptance policy on `internalManagementRoute` is tightened.
