# 1.3.0 — credential cantonment, bootstrap lock, formal wire contract

Minor release. Strictly additive. 1.0.x/1.1.x/1.2.x consumers upgrade without code, wire, or Redis-layout change. Defaults reproduce 1.2.x byte-identical behavior; two new options opt-in to the v1.3.0 enforcement layers.

## What it adds (in one sentence)

- **F1 — Purpose cantonment**: a credential can be tagged `purpose: "propagation-only"`; the lib then refuses to verify it on any path other than the configured `internalManagementRoute`.
- **F2 — Bootstrap-window lock**: an API can refuse every request (signed business calls + non-named management writes) until one specific credential is stored.
- **F3 — Formal wire contract**: `docs/wire-contract.md` becomes the canonical spec; `test/vectors/*.json` certifies any cross-language port.

## New surface

### Types

- `HmacCredentialPurpose = "any" | "propagation-only"` — exported.
- Optional `purpose?: HmacCredentialPurpose` added to:
  - `HmacClientCredential` (read shape)
  - `CreateHmacClientOptions` (passed to `clients.create`)
  - `HmacCredentialWriteOptions` (5ᵗʰ param of `setSecret` / `setSecretHash`)
  - `PropagateHmacClientOptions` (wire payload)
  - `StoredClientCredentialRecord` (persisted record; legacy-tolerant parse)
- Optional `internalManagementRoute?: string` + `requireBootstrapClientId?: string` added to `VerifyHttpSignatureInput`. The HTTP `init.ts` wrapper injects them from the resolved init options on every call.
- Optional `requireBootstrapClientId?: string` added to `InitializeHmacHttpAuthOptions` and `InitializeHmacMessageAuthOptions`.

### Error codes

- `PROPAGATION_ONLY_FORBIDDEN` (HTTP 403): a credential whose stored `purpose` is `"propagation-only"` was used outside the configured `internalManagementRoute` (HTTP track) or for sign/verify on the message track.
- `BOOTSTRAP_LOCKED` (HTTP 403): the API was initialized with `requireBootstrapClientId` and the named credential is not yet stored locally.

### Behavior

- `verifyHttpSignature` enforces F1 after the signature compare + nonce consume: rejection burns the nonce so a misuse attempt cannot retry with another path during the skew window.
- `verifyHttpSignature` enforces F2 before any header parsing: rejection does not leak whether the inbound clientId is known.
- `handleInternalManagementRequest`:
  - keeps `GET` open at all times and emits `bootstrapLocked: boolean` in the body so orchestrators can detect the state.
  - accepts `POST` only when `payload.clientId === requireBootstrapClientId` while locked.
  - refuses `PUT` / `PATCH` / `DELETE` with `BOOTSTRAP_LOCKED` while locked.
- `initializeHmacMessageAuth`:
  - throws `BOOTSTRAP_LOCKED` on `signMessage` and `verifyMessage` while locked.
  - throws `PROPAGATION_ONLY_FORBIDDEN` on either method whenever the stored record carries `purpose: "propagation-only"` (the message track has no path concept, so the credential is refused outright).
- `propagateClientToApis` ships the optional `purpose` field on the payload only when explicitly set; the default wire stays byte-identical to 1.2.x.

## Wire spec & test vectors (F3)

- `docs/wire-contract.md`: the canonical specification of the wire. Lists cryptographic primitives, header names, Redis layout, internal-management route shape, request/response bodies, error codes, and v1.3.0 additions. **Single source of truth** for cross-language ports — the lib and downstream packages (e.g. `@naskot/node-hmac-auth-management`) link to it rather than duplicating.
- `test/vectors/hash-client-secret.json`: 20 deterministic cases of `hashClientSecret(secret, secretToken)`.
- `test/vectors/sign-request.json`: 33 deterministic cases of `signRequest({method, path, timestamp, nonce, body, secret})`.
- `test/vectors/internal-route-flows.json`: 5 multi-step flows covering bootstrap, duplicate-create rejection, update + revert (v1.2.0), bootstrap-locked refusal (v1.3.0), and propagation-only cantonment (v1.3.0).
- `test/wire-vectors.test.ts`: loads every vector and asserts the lib still matches byte-identical. Any drift is a wire break that fails CI.

## Combined value

F1 + F2 are independently usable but reinforce each other when combined:

- Even if the propagation key declared as `requireBootstrapClientId` leaks, declaring it with `purpose: "propagation-only"` means an attacker can only push other (auditable) credentials with it. The business routes stay closed.
- The bootstrap window is no longer a generic attack surface: only the named clientId gets through the first unsigned POST.

F3 unlocks multi-language meshes. A target written in Python / Go / Rust / Java that round-trips every vector with byte-identical output is certified wire-compatible. The orchestrator (`@naskot/node-hmac-auth-management`) can then drive that target without knowing it is not Node.

## Compatibility

- Defaults: `purpose` is absent on the wire / record / option until explicitly set. `requireBootstrapClientId` is `undefined` by default. A 1.2.x consumer that upgrades to 1.3.0 without touching its init code observes zero behavior change.
- Records persisted by 1.0.x / 1.1.x / 1.2.x parse cleanly. Missing `purpose` / `fromDbSeed` defaults to `undefined` / `false` respectively.
- Wire is byte-identical to 1.2.x for any payload that does not explicitly set `purpose`. Targets on 1.0.x / 1.1.x / 1.2.x simply ignore the new field.

## POC

`poc/nest-source` boots, runs the legacy torture suite (62 assertions, unchanged), then runs `runV1_3_0_Demo()` in an isolated Redis namespace. The demo asserts:

1. GET reports `bootstrapLocked: true` while locked.
2. POST `{clientId: "intruder"}` refused with `BOOTSTRAP_LOCKED`.
3. PUT / PATCH / DELETE refused with `BOOTSTRAP_LOCKED`.
4. Bootstrap POST of the named credential releases the lock.
5. Subsequent signed GET reports `bootstrapLocked: false`.
6. Signed POST on a non-management route with a propagation-only credential → `PROPAGATION_ONLY_FORBIDDEN`.
7. Same credential signs the management route just fine.
8. `signMessage` with a propagation-only credential → `PROPAGATION_ONLY_FORBIDDEN`.

Output: one `PASS / FAIL` line per assertion plus a `summary ok=8/8` line.
