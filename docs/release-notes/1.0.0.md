# 1.0.0 — propagation: hash on the wire + Redis fallback

Two behavior changes inside **`propagateClientToApis`** only. Nothing else in the library moves. Signed HTTP requests, message signing/verification, and `clients.create / setSecret / setSecretHash` keep their original `HMAC_SECRET_TOKEN`-dependent semantics. The internal-management bootstrap-then-auth rule is also unchanged.

## Change #1 — propagation transports the hash, not the plain secret

**Before (≤ 0.5.4)** — the payload sent to the target carried the plain `secret`. The target re-hashed it with **its own** `HMAC_SECRET_TOKEN`, so a target with a different token stored a **different** `secretHash`. Subsequent signed requests failed signature verification.

**Since 1.0.0** — the source hashes the secret **locally** with its own `HMAC_SECRET_TOKEN`, and the payload carries that `secretHash` instead. The target stores it as-is via `setSecretHash`. Both Redis stores end up with the **same** `secretHash`. Signed requests verify even if source and target have **different** `HMAC_SECRET_TOKEN` values.

The caller code is unchanged — just pass `secret` exactly as before:

```ts
await hmacAuth.propagateClientToApis({
  operation: "create",
  targets: ["https://target-with-different-token.example.com"],
  clientId: "client_mobile",
  secret: "superSharedSecret", // hashed locally, never sent in clear
  allowedIps: ["0.0.0.0/0", "::/0"],
  apiFetch: signer,
});
```

## Change #2 — `secret` becomes optional when the client is already local

When **both** `secret` and `secretHash` are omitted, the lib looks up `clientId` in the local Redis (the same store that backs `clients.get(...)`). If a record exists, its `secretHash` is reused for propagation.

```ts
// `client_mobile` was already created locally via clients.create / setSecret
// or seeded at boot. No need to repeat the plain secret here:
await hmacAuth.propagateClientToApis({
  operation: "create",
  targets: ["https://target-x.example.com"],
  clientId: "client_mobile", // lib reads its secretHash from local Redis
  // secret / secretHash both omitted
  allowedIps: ["0.0.0.0/0", "::/0"],
  apiFetch: signer,
});
```

If the `clientId` is not in local Redis either, the lib throws (as before) — at least one of `secret`, `secretHash`, or a local record is required.

## Resolution priority

When `propagateClientToApis` builds the outgoing payload, the `secretHash` shipped to the target is resolved in this order:

1. **Explicit `secretHash` from the caller** — full override, used as-is.
2. **Local Redis lookup on `clientId`** — only when both `secret` and `secretHash` are omitted.
3. **`hashClientSecret(secret, localSecretToken)`** — computed locally from the caller-provided plain `secret`.

The plain `secret` is never sent on the wire.

## Out of scope

These changes apply **only** to `propagateClientToApis`. The rest of the library is unchanged:

- `signedHttpFetch`, `verifyHttpSignature`, `verifyHttpRequest`: still produce/verify signatures derived from the local `secretHash` (which itself depends on `HMAC_SECRET_TOKEN`). Two services that did not exchange a credential via the propagation flow still need a coherent setup to sign each other.
- `signMessage`, `verifyMessage`: identical.
- `clients.create`, `clients.setSecret`, `clients.regenerateSecret`: still hash with the local `HMAC_SECRET_TOKEN` at write time.
- Internal-management route bootstrap-then-auth: identical.

## FAQ — propagation acceptance scenarios

Setup: source declares `totoId` in `internalCredentials` (hashed locally with source's `HMAC_SECRET_TOKEN`, stored in source Redis as `secretHash_source`). Later, source adds a `propagationPlans` entry to push `totoId` to a remote API.

> **Q1 — The remote API has zero credentials in Redis. What happens?**
>
> **Accepted.** The internal-management route falls into bootstrap mode (`authRequired = knownClients.length > 0 → false` in `handleInternalManagementRequest`). The POST is processed **without HMAC signature verification**, `totoId` is created in the remote Redis with the source's `secretHash_source` stored as-is. From this point on, the remote has 1 client and any further propagation will require valid HMAC auth.

> **Q2 — The remote API already has at least one credential, but the source has none (cannot sign).**
>
> **Rejected (403 "Internal HMAC management authentication failed").** Two compounding reasons: (a) the source cannot construct a signed propagation request — `clients.get(signerClientId)` returns null on the source side, so the caller throws before any HTTP call leaves the source; (b) even if a request were forged with an unknown signer, the remote runs `verifyHttpSignature`, fails to find that signer in its own Redis (`UNKNOWN_CLIENT`), and the internal handler swallows the error into a 403.

> **Q3 — Both source and remote already have credentials, and the source uses one of its own local credentials as `signerClientId` to propagate `totoId`.**
>
> **Accepted only if the signer's `secretHash` is identical on both sides.** That is the typical case when the signer was previously distributed via the same propagation flow (1.0.0 transports the hash, so both sides end up with byte-identical records). If the signer was seeded independently on each side with different `HMAC_SECRET_TOKEN` values, source and remote hashes diverge → remote `verifyHttpSignature` throws `BAD_SIGNATURE` → handler returns 403. In practice: propagate the signer first (bootstrap path or chain), then every downstream propagation works.

## End-to-end proof in the POC

`poc/` has been updated to demonstrate the new behavior:

- The 3 services (`nest_source`, `nest_target`, `express_target`) ship with **distinct** `HMAC_SECRET_TOKEN` values (`source_token_alpha`, `target_token_beta`, `express_target_token_gamma`).
- `nest_source/src/microservice.cfg.ts` declares 11 propagated `clientId`s, including `client_via_redis_lookup` whose propagation plan **omits** the `secret` (exercising Change #2).
- After boot, `nest_source` signs a `/secure/poc` fetch with each propagated `clientId` against both targets and logs:
  ```
  [nest_source] cross-token verify summary ok=11/11 details=[...]
  ```
  proving each propagated credential is verifiable on both targets despite the token mismatch, including the one that relied on the Redis fallback.
