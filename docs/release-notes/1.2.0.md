# 1.2.0 — dynamic DB-seed origin, TTL backup, REVERT operation

Minor release. Strictly additive. 1.0.x/1.1.x consumers upgrade without code, wire, or Redis-layout change.

## New surface

- Types:
  - `PropagateHmacClientOptions.fromDbSeed?: boolean`
  - `CreateHmacClientOptions.fromDbSeed?: boolean`
  - `HmacClientCredential.fromDbSeed?: boolean` (optional for retrocompat)
  - `HmacCredentialWriteOptions { fromDbSeed?: boolean }` (5th param of `setSecret`/`setSecretHash`)
  - `HmacCredentialRevertResult { reverted: boolean; restoredSecretHash?: string }`
  - `HmacInternalPropagationOperation` += `"revert"`
- Methods: `clients.revert(clientId)` on HTTP store and message bridge.
- Init: `dbSeedBackupTtlSeconds?: number` on `InitializeHmacHttpAuthOptions` and `InitializeHmacMessageAuthOptions` (default 600).
- Route: PATCH accepted on the existing `internalManagementRoute`. Dispatches to `clients.revert`. No new route.
- Wire: `propagateClientToApis({ operation: "revert", targets, clientId, targetStore? })` sends PATCH carrying `{ clientId, kind? }` only.
- Store: `setBackupSecretHash`/`getBackupSecretHash`/`clearBackupSecretHash` on `RedisCredentialStore`. Key shape `<namespace>:credentials-backup:<clientId>`, native Redis TTL. `RedisLikeClient` now requires `get` + `del`.

## Semantics

- Backup TTL is written automatically on `setSecret`/`setSecretHash` only when `options.fromDbSeed === true` AND a record exists AND the hash changes. Static credentials never get a backup.
- `clients.revert(clientId)` restores from backup, clears the backup, preserves `fromDbSeed/allowedIps/expiresAt`. No-op if no backup.
- Wire byte-identical to 1.1.x when `fromDbSeed` is omitted or false.
